# 场景：业务反馈闭环

适用于可以重复运行、观察结果并调整动作的业务过程，例如广告投放、库存补货、定价、销售漏斗和运营实验。

控制论可以帮助管理一个已经定义的业务闭环，但不能证明用户需求存在，也不能替代真实成交、成本和利润证据。

## 典型闭环

```text
目标：在现金和风险边界内改善本轮明确结果，例如验证最终利润
观测：投入、曝光、成交、退款、履约成本和现金变化
动作：停止、调整预算、价格、库存或流程
扰动：竞争、平台规则、季节、供应和用户变化
验证：比较行动前后结果，并记录仍无法解释的变化
```

## 设计重点

1. 区分愿景、业务目标和本轮要验证的具体假设；
2. 观测最终结果，不用点击、访问量或 AI 输出替代利润和用户价值；
3. 提前设定时间、资金、库存和不可逆风险上限；
4. 考虑反馈延迟，例如退款、复购和库存成本可能晚于成交出现；
5. 一次结果只能判断本次事实，重复验证后才能讨论可重复性；
6. 外部变量过多时保留 `unknown`，不要把相关性包装成因果；
7. 目标本身错误时，应停止或换目标，而不是继续优化执行效率。

## 运行合同

开始前写清本轮假设、基线、可用现金、库存、订单与未结算负债等当前状态，以及预算上限、可执行动作、授权人、观察窗口和结算日。涉及付款、调价、投放或库存承诺时，同时规定单次与累计额度；Agent 超额只能提议，不能执行。

利润必须使用预先约定且可复算的口径，例如：

```text
本轮已结算利润 = 已结算收入
               - 商品成本 - 履约成本 - 退款与售后
               - 渠道与支付费用 - 本轮投放费用 - 约定税费
```

未结算订单、预计复购和未确认成本分别标为 `estimated` 或 `unknown`，不能计入 `actual` 利润。观察窗口至少覆盖本业务约定的退款、拒付和结算周期；周期未结束只能给中间状态。

达到损失、权限或数据完整性边界时，安全状态是暂停自动投放、调价、采购和接单，不再新增资金或库存承诺，保留订单与账务证据并由人完成对账。可重复性必须跨预先定义的独立批次、时间窗口或客群达到阈值；此前只能报告本次 `actual`。

本轮最终只使用四类终态：

| 终态 | 判定 |
|---|---|
| `PASS` | 观察窗口结束，目标按约定口径达成且未突破边界 |
| `FAIL` | 证据足以表明目标未达成，或已触发失败条件 |
| `UNKNOWN` | 关键数据缺失、归因不可分或结算周期未结束 |
| `STOPPED` | 因损失上限、权限、安全或用户决定提前停止 |

## 常见失败

- 只优化点击率，却忽略退款和最终利润；
- 看到短期上涨立即放大预算，没有等待完整反馈；
- 同时改变价格、素材、受众和页面，无法判断原因；
- 将一次成交解释成稳定需求；
- 业务失败后用“学到很多”替代结果判断；
- 因为流程可自动化，就默认它值得长期运行。
- 用下单金额、毛利或预计收入冒充已结算利润；
- Agent 在没有额度和授权的情况下调整资金、价格或库存承诺。

## 最小检查

- 当前控制的是业务结果，还是只控制了中间指标？
- 完整反馈需要等待多久？
- 最大允许损失和停止条件是什么？
- 哪个动作能够消除当前最关键的未知？
- 结果能判断哪一个具体假设，不能判断什么？
- 当前数字是 `actual`、`estimated` 还是 `unknown`，结算周期结束了吗？
- 当前动作是否在已授权的单次与累计额度内？

人的偏好、竞争行为和市场需求会因规则而变化。业务系统只能借鉴控制论方法，不能假装具有物理控制系统同等的精确性。
