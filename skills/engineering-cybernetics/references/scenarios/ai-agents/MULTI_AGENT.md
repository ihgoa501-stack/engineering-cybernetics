# 垂直场景：多 Agent 协作

适用于多个 Agent 并行研究、审查、实现、验证、辩论或在不同阶段交接任务。多 Agent 的价值不是“意见更多”，而是在上下文、工具、搜索空间或审查角度确实需要拆分时，提高覆盖率并降低单个上下文的污染。

本场景继承[工程控制论总纲](../../CORE.md)和[AI Agent 与自动化](../AI_AGENTS.md)的规则；本文件只增加多 Agent 协作特有的控制约束。

调研依据：

- Anthropic 将多 Agent 研究系统描述为“主 Agent 规划并创建并行子 Agent 搜索”，适合复杂、开放、路径依赖的研究；但也指出协调、评估和可靠性会变难。
- Anthropic 另文给出三类适合多 Agent 的情况：上下文污染、可并行任务、专业化能改善工具选择或任务聚焦；否则协调成本通常超过收益。
- OpenAI Agents SDK 把多 Agent 编排分成两类：handoff 让专家接管当前分支；agents-as-tools 让主 Agent 保持最终回答所有权。
- LangChain/LangGraph 的经验强调：读多写少更适合并行；多 Agent 写入会带来上下文传递和合并冲突。

---

## 一、什么时候使用

满足至少一项才使用多 Agent：

1. **上下文隔离**：子任务会产生大量材料，但主任务只需要摘要或证据；
2. **并行搜索**：不同方向可以独立探索，互不等待；
3. **专业隔离**：不同 Agent 需要不同工具、规则、领域知识或审查视角；
4. **独立复核**：需要魔鬼代言人、安全审查、QA、事实核查等低写入风险审查。

以下情况优先使用单 Agent：

- 任务需要所有人共享同一细节上下文；
- 多个 Agent 都要同时写同一文件、同一数据库或同一外部对象；
- 目标很小，单 Agent 能直接完成并验证；
- 用户只是想要“更多信心”，但没有新的观测角度或验证方法；
- 成本、时间、权限或停止条件不清楚。

---

## 二、控制目标

启动多 Agent 前，主控 Agent 必须写清：

```text
总目标：最终要交付什么可验证结果？
拆分理由：为什么单 Agent 不够？
Agent 分工：每个 Agent 只负责什么？
共享上下文：必须传递哪些事实，哪些不传？
写入权限：谁能写？谁只能读和建议？
预算上限：最多几个 Agent、几轮、多少工具调用或时间？
合并规则：冲突结论如何裁决？
验收证据：最终结果用什么外部证据验证？
停止条件：什么情况必须停止，不继续加 Agent？
```

“多派几个 Agent 看看”不是控制目标；它只能作为低风险观察动作，不能作为完成证据。

---

## 三、状态、信号与动作

| 控制元素 | 本场景中的含义 |
|---|---|
| 目标状态 | 多个 Agent 的工作被正确拆分、合并，并产生可验证结果 |
| 当前状态 | 总任务、子任务、共享上下文、权限、预算、外部对象和已有证据 |
| 观测信号 | 子 Agent 输出、引用证据、工具结果、diff、测试、成本、冲突点 |
| 控制动作 | 派发、限制、取消、补充上下文、合并、要求证据、升级给用户 |
| 扰动 | 上下文丢失、重复劳动、写入冲突、假共识、成本失控、过期信息 |
| 约束 | 用户授权、工具权限、唯一写入者、预算、时间、安全和隐私 |
| 反馈 | 最终验收、独立验证、冲突闭合、成本是否在上限内 |
| 安全状态 | 停止新增 Agent；冻结写入；保留现场；把结论标记为 `UNKNOWN` |

多 Agent 的输出是观测信号，不是事实本身。事实等级只由证据决定。

## 四、推荐控制结构

先选最弱的架构，只有当前一层无法满足目标、观测或安全边界时才升级。

| 架构 | 适合场景 | 控制要点 |
|---|---|---|
| 单 Agent + 工具 | 目标小、上下文完整、验证简单 | 不拆分；用工具验证真实结果 |
| 主控 Agent + 子 Agent | 并行研究、独立审查、资料收集 | 主控保留最终答复；子 Agent 默认只读 |
| 主控 Agent + 专家工具 Agent | 不同工具或领域隔离明显 | 专家只返回证据和建议，不拥有最终结论 |
| Handoff 交接 | 任务阶段或责任边界清楚 | 同一时刻只有一个负责人 |
| 审查委员会 | 高风险决策前暴露盲点 | 多数意见不等于事实，按证据裁决 |
| Swarm/群聊 | 动态探索且低写入风险 | 必须有状态机、轮次上限和终止条件 |

架构选择不是组织图设计，而是控制面设计：谁看见什么、谁能行动、谁负责合并、谁验证结果、失控时停在哪里。

### 1. 主控-子 Agent

默认结构。主控 Agent 保持最终责任，子 Agent 只交付有限结果。

适合：

- 研究、资料收集、代码审查、方案比较；
- 用户需要一个统一结论；
- 子 Agent 之间不需要互相协调。

控制规则：

- 子 Agent 必须拿到目标、边界、输出格式、停止条件；
- 子 Agent 默认只读，除非明确授权；
- 主控 Agent 负责合并、裁决和最终验证；
- 子 Agent 之间不要互相覆盖结果。

### 2. Handoff 交接

当某个专家应该接管当前分支时使用。交接后，责任从上一个 Agent 转移给下一个 Agent。

适合：

- 客服分流到退款、技术、账户等明确分支；
- 工作阶段明显不同，例如需求澄清后交给实现 Agent；
- 接管方需要不同工具或规则。

控制规则：

- 交接必须说明当前状态、已验证事实、未解决问题和授权边界；
- 交接后只能有一个当前负责人；
- 不能让多个 Agent 同时认为自己拥有最终答复。

### 3. 审查委员会

多个 Agent 只做独立审查，不直接写入。

适合：

- 魔鬼代言人；
- 安全、产品、工程、QA、成本等不同视角审查；
- 高风险决策前暴露盲点。

控制规则：

- 审查 Agent 输出必须标记证据等级；
- 多数意见不等于事实；
- 主控 Agent 只采纳有证据或能降低风险的意见；
- 改变方向、成本或风险时交给用户裁决。

### 4. Swarm/群聊

多个 Agent 互相交谈、动态转移。默认不推荐作为工程起点。

仅在以下条件满足时使用：

- 有清晰的状态机、发言选择或终止规则；
- 每轮都有可观测进展；
- 成本和循环上限明确；
- 失败不会写坏真实外部状态。

## 五、真实场景

### 1. 并行调研

例子：市场、技术方案、竞品、法规、论文、供应商。

控制：

- 每个 Agent 只查一个方向；
- 要求来源链接、日期和不确定性；
- 主控合并重复项并标记冲突；
- 最终结论必须回到原问题，不按材料多少判定完成。

### 2. 代码实现 + 独立审查

例子：一个 Agent 实现，另一个只读 review。

控制：

- 实现 Agent 是唯一写入者；
- 审查 Agent 不直接改文件，只给问题和证据；
- 主控或实现 Agent 修复后跑最小验证；
- 不能用“审查通过”替代测试或真实运行。

### 3. 魔鬼代言人

例子：用户要判断项目方向、产品设计、架构取舍。

控制：

- 魔鬼代言人只负责找反例、失败路径和隐藏成本；
- 不允许把批判本身当成最终结论；
- 主控区分 actual、inferred、unknown；
- 重大方向仍由用户决定。

### 4. 多工具专家

例子：一个 Agent 查 GitHub，一个查网页，一个查本地代码，一个查文档。

控制：

- 每个 Agent 只拿必要工具；
- 禁止把工具权限扩成全权限；
- 工具结果要带来源和时间；
- 主控根据证据而不是角色名裁决。

### 5. 长任务阶段交接

例子：需求分析 -> 实现 -> 测试 -> 发布。

控制：

- 每次交接必须有状态摘要、已完成证据、未完成项、停止条件；
- 下一个 Agent 先验证输入状态，再继续；
- 不允许因为交接丢失用户约束或已有失败。

---

## 六、失败模式

- 子 Agent 指令太短，导致重复搜索、遗漏范围或答非所问；
- 多个 Agent 同时写同一对象，产生冲突；
- 主控没有合并标准，只把多个结论拼接起来；
- 多数 Agent 同意，但没有新的事实证据；
- 为简单任务派太多 Agent，成本高于收益；
- 子 Agent 过度自主，越过用户授权；
- 交接时丢失已验证事实、约束或停止条件；
- 群聊没有终止条件，进入无限讨论。

---

## 七、最小检查

- 为什么必须多 Agent？单 Agent 缺什么？
- 哪个 Agent 拥有最终答复？
- 谁可以写入？谁只能读？
- 每个子 Agent 的输入、输出格式和停止条件是什么？
- 冲突结论用什么证据裁决？
- 成本、轮次和工具调用上限是什么？
- 最终结果如何独立验证？

---

## 八、完成判定

- `PASS`：子任务已按边界完成，冲突已裁决，最终结果有独立证据；
- `FAIL`：多 Agent 输出互相冲突且证据显示目标不成立，或已突破硬边界；
- `UNKNOWN`：证据不足、外部状态不明、成本/权限/加载状态无法确认；
- `STOPPED`：达到预算、风险、权限或用户设定的停止条件。

多 Agent 的正确用法是增加观测覆盖和专业隔离；不是增加表决人数。
