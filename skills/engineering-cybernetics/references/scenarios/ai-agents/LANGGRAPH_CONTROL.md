# 垂直场景：LangGraph 控制场景

适用于已经决定使用 LangGraph 设计 Agent 工作流、状态图、多 Agent 编排、循环、持久化、人类介入或失败恢复的任务。

本场景继承[工程控制论总纲](../../CORE.md)、[AI Agent 与自动化](../AI_AGENTS.md)和[多 Agent 协作](MULTI_AGENT.md)的规则；本文件只增加 LangGraph 实现层特有的控制约束。

LangGraph 是实现工具，不是控制目标。先判断任务是否真的需要状态图；如果单 Agent、普通函数、队列任务或一次性脚本已经能完成并验证，不使用 LangGraph。

调研依据：

- LangGraph 官方文档把它定位为低层 agent orchestration runtime，重点能力包括 durable execution、streaming、human-in-the-loop 和 persistence。
- LangGraph Graph API 的核心元素是 `State`、`Node`、`Edge`：State 表示当前快照，Node 执行动作并返回状态更新，Edge 决定下一步执行哪个节点。
- LangGraph 支持循环工作流，状态会随节点执行持续演化；节点边界也影响 checkpoint、恢复、观测和重试粒度。
- LangChain 多 Agent 文档把 custom workflow 归为使用 LangGraph 混合确定性逻辑与 agent 行为的模式。

---

## 一、什么时候使用

满足至少一项才考虑 LangGraph：

1. **状态会跨多步演化**：任务需要保存中间状态，而不是一次 LLM 调用；
2. **需要循环或条件路由**：不同结果会进入不同节点，失败后可能回到前一步；
3. **需要暂停和恢复**：人类审批、长任务、外部系统等待或失败恢复；
4. **需要多 Agent 编排**：主控、router、handoff、subgraph、supervisor 或 swarm；
5. **需要可观测执行路径**：必须知道每一步为什么走到下一个节点；
6. **需要持久化记忆**：运行中断后要从检查点继续，而不是从头重跑。

以下情况优先不用 LangGraph：

- 一次性问答、简单 RAG、单次工具调用；
- 只有两个固定步骤，没有循环、暂停或恢复；
- 业务状态无法明确定义；
- 没有测试、trace 或日志来验证图是否按预期运行；
- 团队还没搞清楚节点、状态、边界和失败处理，只是想“显得架构高级”。

## 二、控制目标

设计 LangGraph 前，先写清：

```text
目标状态：图最终要把任务带到什么可验证结果？
State：哪些字段代表真实状态？哪些只是临时上下文？
Node：每个节点只做哪一类动作？
Edge：每条边根据什么条件路由？
Loop：哪些循环允许发生，最多几次？
Interrupt：哪些动作前必须暂停给人确认？
Checkpoint：在哪些边界保存状态，失败后从哪里恢复？
Observation：如何追踪每次节点输入、输出、路由和错误？
Safe state：图失控、证据不足或外部状态不明时停在哪里？
```

“把流程画成图”不是目标。目标是让状态、动作、路由、反馈和停止条件都可检查。

---

## 三、状态、信号与动作

| 控制元素 | LangGraph 中的含义 |
|---|---|
| 目标状态 | Graph 到达 `END` 或安全暂停点，且验收证据成立 |
| 当前状态 | `State` 字段、消息历史、检查点、活跃节点、外部副作用记录 |
| 观测信号 | 节点输入输出、edge 选择、checkpoint、trace、日志、测试、人工审批 |
| 控制动作 | 执行节点、更新状态、路由、重试、暂停、恢复、回滚计划、终止 |
| 扰动 | LLM 不稳定、工具失败、重复执行、并行状态合并、人工延迟、外部系统变化 |
| 约束 | 幂等性、权限、成本、循环上限、隐私、唯一写入者、人工确认边界 |
| 反馈 | 节点级测试、端到端图测试、trace 审查、真实外部状态验证 |
| 安全状态 | `interrupt` 等待人工输入；或进入错误节点并停止外部写入 |

LangGraph 的 `State` 不是随手塞上下文的袋子。它是控制系统的状态面，字段越乱，路由越不可控。

## 四、架构选择

先选最弱的图结构，只有当前结构无法满足控制要求时才升级。

| 图结构 | 适合场景 | 控制要点 |
|---|---|---|
| 线性图 | 固定步骤，失败代价低 | 每步独立验证，失败即停 |
| 条件路由图 | 分类、分流、不同工具路径 | 路由条件必须可观测，不靠模糊叙述 |
| 循环图 | 搜索-评估-补充、生成-检查-修复 | 必须有最大轮次和停止条件 |
| 人类介入图 | 审批、资金、生产、外部发送 | 高风险节点前暂停，不先执行后汇报 |
| Supervisor 图 | 主控调度多个专家 Agent | 主控拥有最终状态和最终裁决 |
| Handoff 图 | 专家接管当前分支 | 同一时刻只有一个当前负责人 |
| Subgraph 图 | 某个复杂子流程可独立验证 | 子图有自己的输入、输出和失败边界 |
| Swarm 图 | 动态专家互相交接 | 仅用于低写入风险探索，必须限制轮次 |

## 五、LangGraph 特有控制规则

### 1. State 要小而硬

- 只放路由和恢复必须依赖的字段；
- 区分事实、推断、草稿、外部副作用；
- 对会累计的字段使用明确 reducer；
- 不把整段长上下文无边界塞进共享状态。

### 2. Node 要有单一失败模式

- 一个节点只混合一种主要副作用；
- 外部 API、数据库写入、邮件发送、付款、部署等动作独立成节点；
- 失败代价不同的动作不要合并在同一节点；
- 节点输出必须是状态更新，不是口头总结。

### 3. Edge 是控制逻辑，不是装饰

- 每条条件边必须对应可检查字段；
- 不允许“模型觉得差不多”就路由到高风险节点；
- 默认边应该走向安全状态，而不是继续执行；
- 路由函数要可测试。

### 4. Loop 必须有刹车

- 每个循环记录轮次、成本和上次失败原因；
- 重试只用于瞬时失败；
- LLM 可修复错误回到修复节点；
- 权限、资金、生产、数据破坏风险进入人工暂停；
- 超过上限进入 `UNKNOWN` 或 `STOPPED`，不无限循环。

### 5. Interrupt 放在高风险动作之前

人类介入的意义是阻止错误动作发生，不是让人事后阅读事故报告。

必须暂停的例子：

- 对外发送消息；
- 付款、退款、下单、调价；
- 删除或覆盖重要数据；
- 生产发布或权限变更；
- 目标、成本或风险发生显著变化。

### 6. Checkpoint 不是幂等性

持久化能帮助恢复状态，但不能自动保证外部副作用只发生一次。

对外部写入节点必须记录：

- 幂等键；
- 外部请求 ID；
- 写入前后的真实状态；
- 恢复后是否已经执行过；
- 不确定时进入人工确认。

### 7. Trace 是反馈，不是摆设

每次真实运行至少能回答：

- 哪个节点决定了下一步？
- 当时 State 是什么？
- 哪个证据触发了路由？
- 哪个外部动作已经发生？
- 为什么停止、重试、暂停或结束？

## 六、真实场景

### 1. 客服/运营工单流

流程：分类 -> 查资料 -> 草拟回复 -> 高风险判断 -> 人工审批 -> 发送。

控制：

- 分类结果进入 State；
- 发送节点前必须有人工审批；
- 外部发送节点必须幂等；
- 无法确认用户身份或政策依据时停止。

### 2. 代码修复流

流程：复现 -> 定位 -> 修改 -> 测试 -> 审查 -> 总结。

控制：

- 测试失败可循环回修改；
- 最大修复轮次明确；
- 审查 Agent 只读；
- 不能用模型总结替代测试结果。

### 3. 研究报告流

流程：拆问题 -> 并行搜索 -> 来源评估 -> 冲突合并 -> 输出。

控制：

- 每条结论绑定来源；
- 搜索子图只返回证据，不直接下结论；
- 冲突无法裁决时标记 `UNKNOWN`；
- 不按资料数量判断完成。

### 4. 业务自动化流

流程：读取订单/库存 -> 判断异常 -> 草拟动作 -> 人工确认 -> 执行 -> 验证。

控制：

- 资金、库存、价格、订单动作前暂停；
- 执行后读取真实外部状态；
- 外部结果不明时不重复执行；
- 进入 `UNKNOWN` 并要求人工确认。

### 5. 多 Agent Handoff 流

流程：Router -> 专家 Agent -> 必要时交给另一个专家 -> 主控汇总。

控制：

- handoff 更新当前负责人字段；
- 同一时刻只有一个 Agent 可写最终回复；
- 交接必须带已验证事实和未解决问题；
- 循环交接超过上限停止。

---

## 七、最小检查

- 这个任务为什么不能用普通函数或单 Agent 完成？
- State 里哪些字段是路由必需？
- 哪些节点有外部副作用？
- 哪些边会进入高风险动作？
- 每个循环最多几次？
- 人应该在哪个动作之前介入？
- 崩溃恢复后如何避免重复执行外部写入？
- 最终结果如何用 trace、测试或外部状态验证？

---

## 八、完成判定

- `PASS`：图结构、State、节点、边、循环、暂停点和验证方式都能解释并通过最小测试；
- `FAIL`：图会无限循环、重复外部副作用、越权执行或无法验证结果；
- `UNKNOWN`：加载、持久化、外部状态、trace 或恢复行为无法确认；
- `STOPPED`：发现不需要 LangGraph、范围过大、权限不足或需要用户决定。

LangGraph 的价值是把 Agent 行为变成可观察、可暂停、可恢复的状态图；不是把简单流程包装成复杂架构。
