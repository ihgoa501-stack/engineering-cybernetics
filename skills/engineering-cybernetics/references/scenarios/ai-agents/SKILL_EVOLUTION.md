# 垂直场景：Skill 与规则演化

适用于新增或修改 Agent Skill、Hook、场景文档、项目规则、全局规则、路由器提示词或其他会影响未来 Agent 行为的材料。

本场景继承[工程控制论总纲](../../CORE.md)和[AI Agent 与自动化](../AI_AGENTS.md)的规则；本文件只增加“规则系统如何长期演化”的约束。核心原则是：规则本身也是被控对象，必须用真实失败和真实反馈来调整，而不是因为想象中的未来复杂度不断加规则。

---

## 一、什么时候使用

出现以下情况时加载本场景：

- 用户要新增、修改或删除 Skill、Hook、场景、指令、ETHOS、AGENTS 或类似规则文件；
- 旧路径、旧安装副本、旧 hook 输出或旧文档与当前仓库不一致；
- Agent 反复误用规则，例如过度扩范围、误判场景、漏验证、把建议当授权；
- 多个规则源发生冲突，需要判断优先级、合并或拆分；
- 想把一次真实问题沉淀成以后可复用的场景。

以下情况通常不需要完整闭环：

- 只修 typo、链接或格式；
- 一次性解释规则含义，不改变规则文件；
- 已有场景能完全覆盖，只需要按现有场景执行。

---

## 二、控制目标

每次规则演化前，先写清楚：

```text
触发事实：具体发生了什么问题或新增了什么真实场景？
目标行为：以后 Agent 应该在什么情况下做什么？
保持不变：哪些已有规则、加载路径和安全边界不能破坏？
最小改动：改一个现有场景、加一条触发条件，还是必须新增文档？
验收证据：如何证明新规则能被找到、能被加载、不会制造冲突？
停止条件：什么情况下不继续加规则，只记录 unknown 或交还用户？
```

“以后可能会用到”“感觉更完整”“让 Agent 更聪明”不是足够的控制目标。它们容易把规则系统推向膨胀。

---

## 三、状态、信号与动作

| 控制元素 | 本场景中的含义 |
|---|---|
| 目标状态 | 未来 Agent 能在正确场景下加载最小必要规则，并产生可验证行为 |
| 当前状态 | 仓库规则文件、Skill 安装副本、Hook 输出、场景索引、已有用户反馈 |
| 观测信号 | 真实误用案例、命令输出、加载日志、diff、测试、用户纠正 |
| 控制动作 | 修改现有规则、增加触发条件、新增场景、合并重复规则、删除过期说法、停止 |
| 扰动 | 平台更新、模型行为变化、本地缓存过期、多 Agent 假共识、旧记忆、文档漂移 |
| 约束 | 上级指令优先级、用户授权范围、上下文成本、安全边界、兼容性 |
| 反馈 | 新对话是否能加载、hook 是否仍轻量、场景选择是否更清楚、验证是否通过 |
| 安全状态 | 不改核心规则；保留现状；标记 unknown；把候选场景放入待验证清单 |

规则文件的存在不是生效证据。实际加载路径、hook 输出、安装缓存和新任务行为才是反馈。

## 四、真实生活中的高频场景

### 1. 规则越来越多，Agent 反而更乱

症状：

- 同一件事在多个文件里重复说；
- Agent 不知道该听 ETHOS、AGENTS、Skill 还是 Hook；
- 每次失败都新增一条规则，但没有删除或合并旧规则。

控制动作：

- 先找现有规则是否已经覆盖；
- 能改旧规则就不新增文件；
- 每条规则必须有触发条件和停止条件；
- 高层文件只保留原则，细节下沉到条件加载的参考文档。

### 2. Hook、Skill、插件职责混淆

症状：

- 以为 hook 会自动替代 Skill；
- 把完整方法论塞进 hook，导致每次对话都加载过多内容；
- 新对话里看不到预期提示，但不知道是未安装、未信任、路径错还是缓存旧。

控制动作：

- Hook 只做轻量路由和提醒；
- Skill 放完整执行方法；
- 场景文档放垂直细节；
- 修改后按“源仓库提交 -> 本地安装/同步 -> 新对话验证”闭环检查。

### 3. 多 Agent 一致，但结论仍可能错

症状：

- 多个 Agent 都同意新增规则；
- 没有人检查真实加载路径、实际 diff 或用户原始问题；
- 一致意见被当成事实证明。

控制动作：

- 把多 Agent 结论降级为参考信号；
- 至少用一个独立证据验证真实状态；
- 对写入对象指定唯一写入者，其他 Agent 只做审查；
- 如果证据不足，判定 `UNKNOWN`，不判定 `PASS`。

### 4. 场景增加后没有入口，等于没有生效

症状：

- 新增了场景文件，但没有登记到场景索引；
- 触发条件太抽象，Agent 不知道什么时候读；
- 文件路径变化后旧文档仍引用旧位置。

控制动作：

- 每个新场景必须在 `SCENARIOS.md` 登记触发条件和路径；
- 触发条件写成用户真实会说的话或任务真实会出现的对象；
- 修改后检查链接、路径和最小加载链路。

### 5. 把一次失败扩大成永久复杂度

症状：

- 为一个低频、低风险问题新增大型流程；
- 新规则要求所有任务都做更多检查；
- 没有区分 actual、inferred、unknown。

控制动作：

- 先判断失败代价和复现概率；
- 低风险低频问题优先写成候选经验，不进入默认路径；
- 只有影响授权、安全、费用、真实外部状态或反复误用的问题，才进入强规则。

---

## 五、新增场景的准入标准

只有同时满足以下条件，才新增独立场景文档：

1. 目标状态不同：它要控制的结果不是已有场景的子集；
2. 观测信号不同：需要看不同证据才能判断成败；
3. 控制动作不同：需要不同动作才能把系统拉回目标；
4. 安全边界不同：失败代价、授权边界或停止条件不同；
5. 触发条件清楚：未来 Agent 能从用户请求或任务对象中识别它。

如果只是不同行业里的同一种闭环，优先在已有场景里增加例子，不新增场景。

---

## 六、最小修改模板

```text
actual：观察到的真实问题是什么？
inferred：这个问题说明哪条规则缺失、冲突或过重？
change：本次最小修改是什么？
scope：它只影响哪些场景？明确不影响哪些场景？
validation：如何验证加载、链接、测试或新对话行为？
unknown：哪些仍不知道，不能包装成已解决？
```

---

## 七、完成判定

- `PASS`：规则修改已落到唯一来源，入口已登记，路径有效，必要验证通过；
- `FAIL`：修改后链接、加载、测试或行为验证失败；
- `UNKNOWN`：无法确认真实加载、安装缓存或新对话行为；
- `STOPPED`：发现规则冲突、权限不足、范围扩大或需要用户决策。

规则演化的目标不是“写更多规则”，而是让未来 Agent 在更少歧义下做更正确、可验证、可停止的动作。
