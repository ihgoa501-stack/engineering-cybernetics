# 垂直场景：从真实问题到首个可用闭环

适用于新软件项目，或已经产生很多代码和功能、但还没有完成第一次真实交付的项目。

本场景继承[工程控制论总纲](../../CORE.md)和[软件项目开发](../SOFTWARE_DEVELOPMENT.md)的规则。它只控制第一轮：让一个明确用户能够从起点走到有价值的结果，并产生足以决定下一步的真实反馈。

---

## 一、本轮控制目标

第一轮不是建设完整产品，而是证明一个具体流程能够真实运行和使用。

```text
一个明确用户
    在一个真实场景中
    完成一个端到端流程
    得到一个可观察的结果
    并提供能够改变下一步决定的反馈
```

第一轮需要做完整的是已经选定的流程、必要错误处理和风险边界，不是相邻功能、平台能力或未来扩展结构。

## 二、开始前控制卡

```markdown
### 首个可用闭环控制卡

- 目标用户：
- 具体问题：
- 当前替代方案：
- 价值指标：
- 对比基线：
- 观察窗口：
- 流程起点：
- 流程终点：
- 用户获得的结果：
- 本轮不做：
- 时间上限：
- 资金上限：
- 通过条件：
- 失败条件：
- unknown 条件：
- 停止条件：
- 通过后动作：
- 失败后动作：
- unknown 后动作：
- 重复验证阈值：
- 小规模扩大阈值：
```

无法填写目标用户、具体问题、流程终点和通过条件时，先调查，不进入大规模开发。无法预先定义重复或扩大阈值时，默认结果只能用于学习和下一轮验证，不能扩大。

### 已有大量代码时的接管

进入本轮前先冻结功能扩建，并以选定流程重新划定项目边界：

1. 找出用户真实进入系统的唯一入口、流程终点及直接依赖；
2. 把现有内容标记为“本轮必需”“保持不动”或“与本轮无关”；
3. 为选定流程记录当前能走到哪里、在哪里失败的基线；
4. 未选中的功能默认不修复、不集成、不重构，也不要求为它们完成全量兼容；
5. 只有共享依赖会阻断选定流程或造成数据、安全风险时，才进入本轮。

历史代码量不构成本轮范围。接管目标是隔离一个可交付闭环，不是顺带收拾整个项目。

---

## 三、阶段与阶段门

项目一次只推进当前阶段。完成活动不等于通过阶段门，必须出现对应证据。

| 阶段 | 要完成的事 | 进入下一阶段的证据 |
|---|---|---|
| 1. 问题确认 | 确认用户、场景、痛点和现有替代方案 | 至少有直接观察、真实经历或目标用户表达，并记录当前替代方案的对比基线 |
| 2. 闭环定义 | 划定起点、终点、必要步骤和不做范围 | 能用一句话描述完整流程，并写出价值指标、观察窗口、通过与停止条件 |
| 3. 最小实现 | 只建设跑通该流程所需内容 | 在接近真实环境中从起点走到终点，关键失败不会损坏数据或失去控制 |
| 4. 交付验证 | 让目标用户独立或在最少协助下使用 | 用户实际完成流程，并在预设窗口内观察价值指标；否则留下具体失败位置和原因 |
| 5. 结果判断 | 对比基线，区分事实、原因推断和未知 | 按预先绑定的结果条件和动作选择停止、修正、重复或小规模扩大 |

不得因为开发速度快而跳过问题确认，也不得因为测试通过而跳过真实使用。

---

## 四、项目状态、信号与动作

| 控制元素 | 本场景中的含义 |
|---|---|
| 目标状态 | 一个目标用户完成选定流程并得到结果 |
| 当前状态 | 项目所处阶段、核心阻塞、剩余预算和未决未知 |
| 观测信号 | 访谈或实际经历、运行结果、测试、用户操作和最终结果 |
| 控制动作 | 调查、缩小范围、实现、修复、交付、停止或重复验证 |
| 扰动 | 新想法、竞品功能、AI 建议、技术问题、用户行为变化 |
| 约束 | 时间、资金、数据、安全、能力和用户可承受成本 |
| 反馈 | 用户是否完成流程、是否获得价值、为何继续或拒绝 |
| 安全状态 | 停止扩建，保留可运行成果和证据，重新选择最小下一步 |

代码、文档和测试是项目状态的信号，但不是最终用户结果。

---

## 五、范围控制器

新需求或已有功能只有满足以下至少一项，才能进入本轮：

1. 没有它，选定流程无法从起点走到终点；
2. 没有它，真实用户无法开始或完成使用；
3. 没有它，会产生不可接受的数据、安全、资金或恢复风险；
4. 它能够直接消除当前阻止下一阶段的关键未知。

否则记录到后续清单，不立即实现。竞品拥有、未来可能需要、技术上顺手、AI 建议完整化，都不是进入本轮的充分理由。

如果新增内容使时间、资金或复杂度超过控制卡上限，先删除或替换已有范围，而不是自动扩大上限。

---

## 六、进展信号

### 有效进展

- 原本未知的用户问题得到直接证据；
- 核心流程少了一个真实阻塞；
- 系统在更接近真实环境中跑通；
- 目标用户完成了更多实际步骤；
- 失败位置和原因变得更具体；
- 新证据改变了继续、修改或停止的决定。

### 伪进展

- 代码行数、文件数或功能数增加；
- AI 输出了更多方案或文档；
- 学习时间增加，但没有改变下一步决定；
- 测试数量增加，但核心流程仍无法使用；
- 架构更通用，但第一位用户仍不能完成任务；
- 用“以后有用”解释当前范围扩张。

项目反馈面板只保留能够改变决策的信号。

---

## 七、需要抑制的正反馈

| 失控循环 | 抑制动作 |
|---|---|
| 方向不清 → 增加研究 → 出现更多方向 → 继续研究 | 只调查当前最关键未知，并设置截止时间 |
| 核心流程受阻 → 建相邻功能 → 系统更复杂 → 更难跑通 | 回到唯一端到端流程，删除非必要范围 |
| AI 生成更快 → 改动更多 → 人无法判断 → 继续依赖 AI 扩建 | 缩小单次任务，用真实运行和差异审计恢复可观测性 |
| 测试通过 → 认为已经完成 → 延迟真实交付 | 把目标用户实际使用设为独立阶段门 |
| 一次好结果 → 立即扩大 → 变量增加 → 无法判断原因 | 保持流程不变，先重复验证 |
| 结果不好 → 用学习解释成功 → 不停止 | 分开判断业务结果和学习结果，按预设条件裁决 |

---

## 八、验证与结果判断

第一轮至少需要三层证据，层级不能相互替代：

1. **技术证据**：核心流程在目标环境中可以运行，关键边界有检查；
2. **使用证据**：目标用户实际尝试，能够完成或暴露具体失败位置；
3. **结果证据**：在预设观察窗口内，价值指标相对基线或当前替代方案是否发生预期变化，以及这能判断哪个具体假设。

“用户完成了流程”只属于使用证据，不自动证明流程有用。没有价值指标、对比基线或完整观察窗口时，结果必须标记为 `unknown`，不得据此扩大。

报告结果时区分：

- `actual`：实际发生和直接验证的事实；
- `inferred`：根据事实作出的原因判断；
- `estimated`：尚未由实际数据确认的估算；
- `unknown`：当前证据无法判断；
- `quoted`：用户或来源明确表达的说法。

一次使用只证明该次使用发生；一次成功只证明本次闭环成立。重复验证后才能讨论可重复性，小规模扩大后仍成立才能讨论规模化。

---

## 九、停止、修正与扩大

结果冲突时按固定顺序裁决：

1. 安全、权限、数据和不可逆风险先否决；
2. 时间或资金达到停止上限，本轮立即停止；可修复阻塞不能自动延长预算；
3. 未达到停止线时，再按预设的通过、失败和 `unknown` 条件判断；
4. 执行控制卡预先绑定的动作，不在看到结果后临时降低门槛；
5. 开启新一轮必须重新设定范围、预算和阶段门。

### 停止

- 达到时间或资金上限，仍无法跑通核心流程；
- 用户问题没有获得直接证据，或目标用户不愿尝试；
- 必要成本、风险或复杂度超过当前可承受范围；
- 继续开发不能消除当前关键未知。

### 修正

- 问题存在，但流程起点、终点或交付方式不对；
- 用户尝试后暴露了具体、可修复的阻塞；
- 技术实现可行，但用户成本过高或结果不清楚。

### 重复验证

- 本次闭环成立，但原因或可重复性仍不清楚；
- 外部变量太多，无法判断哪个动作产生结果。

重复次数和观察窗口在开始前写入控制卡。未达到预设阈值时，只能继续验证或停止，不能扩大。

### 小规模扩大

只有同一核心流程达到控制卡预设的重复次数、价值阈值和观察窗口，且当前瓶颈明确来自容量或相邻必要步骤时，才增加用户量、场景或功能。一次只扩大一个主要变量。

## 十、与 AI 编码 Agent 的关系

本文件决定“当前项目应该解决哪一个问题、做到哪里、凭什么继续”。[AI 编码 Agent](../ai-agents/CODING_AGENT.md)决定“其中一次编码任务如何安全、正确地完成”。

项目控制器生成有边界的编码任务；编码任务的验证结果返回项目控制器。编码 Agent 无权因为实现方便而改变目标用户、产品范围、资金上限或阶段门。
