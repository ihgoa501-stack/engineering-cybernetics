# 垂直场景：非技术用户的软件建设控制

适用于一个不懂技术、架构、编程语言、框架和专有名词的人，想用 AI、开发者或团队做一个软件系统、自动化工具、网站、App、Agent 或内部系统。

本场景继承[工程控制论总纲](../../CORE.md)和[软件项目开发](../SOFTWARE_DEVELOPMENT.md)的规则；本文件只增加“非技术用户如何控制技术建设”的约束。

核心原则：非技术用户不需要亲自选择技术细节，但必须控制目标、边界、风险、验收和停止条件。技术方负责推荐默认方案，并用普通语言解释为什么这个方案能服务当前闭环。

---

## 一、什么时候使用

出现以下情况时加载本场景：

- 用户说“我想做一个系统/平台/工具/App/Agent”，但无法判断怎么实现；
- 用户被要求选择技术栈、架构、数据库、框架、语言、云服务或 Agent 框架；
- 技术方案里出现很多名词，但用户不知道它们和结果有什么关系；
- AI 或开发者建议微服务、分布式、LangGraph、AutoGen、RAG、向量库、队列、Kubernetes 等复杂方案；
- 项目已经有代码，但用户不知道现在是否真的可用；
- 用户担心“以后扩展”，因此被推动提前做复杂架构。

以下情况优先不用完整本场景：

- 用户只是问一个技术名词是什么意思；
- 已有明确验收条件的一次小修复；
- 技术选择已经被现有项目、团队或平台强约束；
- 只是在学习技术，不打算交付真实系统。

---

## 二、控制目标

每轮技术建设前，先固定：

```text
用户目标：谁要在什么场景下完成什么事？
结果证据：什么行为证明系统真的有用？
本轮范围：只做哪一个端到端流程？
不做范围：哪些功能、平台化、扩展性暂时不做？
技术默认：推荐用什么技术，为什么足够？
架构边界：系统先简单到什么程度，什么证据才允许升级？
术语解释：每个关键技术名词和用户结果有什么关系？
验收方式：非技术用户如何看见、操作或验证？
停止条件：成本、复杂度、风险或未知到什么程度就停？
```

“技术上更先进”“以后可能需要”“大家都这么做”不是控制目标。

---

## 三、状态、信号与动作

| 控制元素 | 本场景中的含义 |
|---|---|
| 目标状态 | 非技术用户能判断系统是否接近真实可用，而不被技术细节牵着走 |
| 当前状态 | 需求、原型、代码、技术方案、架构图、成本、风险、术语和未知 |
| 观测信号 | 可操作演示、端到端运行、测试、截图、日志、用户操作、成本估算 |
| 控制动作 | 翻译术语、缩小范围、选择默认技术、做 spike、审查架构、停止扩建 |
| 扰动 | 术语包装、AI 过度设计、开发者偏好、框架潮流、未来扩展焦虑 |
| 约束 | 用户理解能力、时间、资金、维护能力、数据安全、交付窗口 |
| 反馈 | 用户能否完成流程、是否看懂取舍、是否能用证据裁决下一步 |
| 安全状态 | 冻结新增技术和功能；保留最小可运行版本；只消除一个关键未知 |

非技术用户的控制面不是代码细节，而是结果、边界、证据、成本和风险。

---

## 四、角色边界

### 用户必须决定

- 要解决谁的什么问题；
- 第一版只完成哪个流程；
- 可接受的时间、资金和风险上限；
- 哪些外部动作、数据、资金、生产环境需要确认；
- 看到什么证据后继续、修改或停止。

### 技术方必须决定并解释

- 默认技术栈；
- 数据如何存；
- 部署在哪里；
- 如何验证；
- 哪些复杂架构现在不需要；
- 什么信号出现后才升级架构。

不要把非技术用户推到“React 还是 Vue”“Postgres 还是 MongoDB”“微服务还是单体”这类选择上。除非选择会直接改变成本、风险、交付时间或用户结果，否则技术方给推荐方案。

---

## 五、技术名词控制

每个关键技术名词第一次出现时，必须翻译成四句话：

```text
它是什么：一句普通话解释。
为什么现在要管：和当前目标有什么关系？
如果不用会怎样：本轮会不会失败？
怎么验证：非技术用户能看到什么证据？
```

如果一个名词无法回答这四项，默认它暂时不进入本轮决策。

### 常见名词的控制问题

| 名词 | 非技术用户要问的问题 |
|---|---|
| 前端 | 用户实际看到和操作的地方是什么？ |
| 后端 | 它负责保存、计算或调用哪些真实东西？ |
| 数据库 | 现在必须长期保存什么数据？没有它会怎样？ |
| API | 哪两个系统需要互相传话？传什么？失败怎么办？ |
| 架构 | 当前复杂度解决了哪个真实风险？ |
| 框架 | 为什么用它比普通实现更快、更稳或更便宜？ |
| 云服务 | 花费、权限、数据和停机风险在哪里？ |
| 队列 | 当前是否真的有异步、排队或失败重试问题？ |
| 向量库/RAG | 是否真的需要从大量材料中检索，而不是普通搜索或数据库？ |
| 微服务 | 是否已经有团队、部署、扩展或故障隔离证据支持拆分？ |
| Agent | 它会观察什么、能做什么、如何停止和验证？ |

## 六、语言、框架和技术栈控制

默认规则：

1. 先使用项目已有技术；
2. 没有已有项目时，选成熟、常见、资料多、部署简单、AI 和开发者容易维护的技术；
3. 不为“以后可能扩大”提前选择复杂栈；
4. 不让用户在等价技术名词之间做伪选择；
5. 只有当技术选择影响成本、交付、风险或长期维护时，才升级给用户裁决。

### 技术选择说明必须包含

```text
推荐方案：
为什么适合当前闭环：
替代方案：
不选替代方案的原因：
当前代价：
未来升级信号：
如果选错，如何回退或替换：
```

### 语言和框架的裁决信号

| 信号 | 控制动作 |
|---|---|
| 已有代码库 | 优先沿用，除非它阻断当前闭环 |
| 目标平台明确 | 使用平台最顺的语言和框架 |
| 团队/维护者明确 | 选维护者能长期处理的技术 |
| 需要快速原型 | 选开发和部署最快的成熟方案 |
| 涉及资金、隐私、生产 | 选可审计、可测试、权限边界清楚的方案 |
| 只是技术偏好 | 不作为决策依据 |

## 七、架构控制阶梯

从最低层开始，只有出现证据才升级。

| 层级 | 架构 | 适合情况 | 升级信号 |
|---|---|---|---|
| L0 | 手工流程、表格、无代码原型 | 还不知道问题是否真实 | 有用户愿意重复使用 |
| L1 | 单应用 + 简单数据库 | 第一个可用闭环 | 核心流程跑通并开始重复验证 |
| L2 | 模块化单体 | 功能增加但仍由一个小团队维护 | 模块边界稳定、改动互相影响明显 |
| L3 | 队列/异步任务/缓存 | 有真实延迟、重试、吞吐问题 | 同步处理已阻塞用户或系统 |
| L4 | 多服务/微服务 | 多团队、多部署、独立扩展或故障隔离 | 单体已成为真实瓶颈 |
| L5 | 分布式平台/复杂编排 | 高规模、高可靠性或长期自动化 | 有运行数据证明简单方案不够 |

默认停在能跑通当前闭环的最低层。架构升级必须绑定实际信号，不绑定想象。

---

## 八、技术方案阶段门

### 1. 需求翻译门

通过条件：

- 用户问题和端到端流程能用普通话讲清；
- 技术名词不会替代用户结果；
- 不做范围已经列出。

未通过时，只做访谈、草图、手工模拟或低成本原型。

### 2. 技术默认门

通过条件：

- 技术方给出一个推荐默认方案；
- 说明为什么足够当前闭环；
- 说明不选复杂方案的理由；
- 说明未来升级信号。

未通过时，不进入大规模开发。

### 3. 架构审查门

通过条件：

- 每个组件都有当前必要性；
- 每条复杂链路都有失败处理；
- 用户能看懂成本和风险；
- 没有为了术语、趋势或未来想象增加复杂度。

未通过时，删除或合并组件。

### 4. 可用性验收门

通过条件：

- 非技术用户能从入口走到结果；
- 关键失败不会损坏数据或失去控制；
- 有截图、日志、测试或真实操作证据；
- 仍然不知道的部分被标记为 `unknown`。

未通过时，不能用“架构已经完成”判定项目完成。

## 九、常见失控

| 现象 | 控制诊断 | 抑制动作 |
|---|---|---|
| 用户被迫选技术栈 | 技术责任被转嫁给非技术用户 | 技术方推荐默认方案并解释取舍 |
| 架构图越来越复杂 | 架构替代了可用结果 | 回到唯一端到端流程 |
| AI 推荐很多框架 | 动作增益过高，观测不足 | 只保留解决当前阻塞的技术 |
| “以后扩展”驱动开发 | 未来假设替代当前证据 | 写升级信号，不提前实现 |
| 术语很多但用户不会用 | 解释没有转化为控制权 | 每个术语绑定验证方法 |
| 测试通过但用户不会操作 | 技术反馈替代使用反馈 | 加入口、演示和用户操作验收 |
| 开发者说“这很简单”但无法展示 | 叙述替代运行证据 | 要可操作 demo、日志或测试 |

---

## 十、最小检查

- 这个系统要让谁完成什么事？
- 第一版只跑通哪一个流程？
- 哪些技术名词会影响成本、风险或验收？
- 技术方推荐的默认方案是什么？
- 为什么不用更复杂架构？
- 什么信号出现后才升级架构？
- 非技术用户如何亲眼验证可用？
- 继续开发、修改、停止分别由什么证据触发？

---

## 十一、完成判定

- `PASS`：用户目标、技术默认、架构边界、术语解释和验收证据都清楚，当前闭环可运行；
- `FAIL`：技术方案无法支撑当前流程，或复杂度/成本/风险已突破边界；
- `UNKNOWN`：无法判断技术可行性、成本、风险、维护能力或用户是否能使用；
- `STOPPED`：用户无法控制方向、技术复杂度不可解释、预算到线或需要用户重新决定目标。

非技术用户做系统时，技术控制的目标不是让用户变成工程师，而是让每个技术决定都能回到普通人可理解的结果、代价、风险和证据。
