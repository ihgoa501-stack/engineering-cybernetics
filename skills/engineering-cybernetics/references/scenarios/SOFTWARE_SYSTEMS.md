# 场景：软件系统

适用于在线服务、数据库、缓存、消息队列、任务调度和分布式系统。这里的控制对象不是机器位置，而是延迟、吞吐、错误率、队列深度、资源消耗和可用性。

当系统会持续运行、状态随负载或依赖变化、且需要在故障后恢复时使用本场景。一次性、低风险、没有持续状态和运行操作的软件任务不适用；项目能否交付由[软件项目开发](SOFTWARE_DEVELOPMENT.md)控制。

## 典型闭环

```text
示例目标：指定用户路径在约定统计窗口内 P99 延迟低于 200ms，同时满足成功率、样本量和成本边界
观测：延迟、错误率、队列深度、资源使用率
动作：扩缩容、限流、熔断、降级、流量切换
验证：动作后用户请求是否恢复，成本是否仍在边界内
```

## 设计重点

1. 用可验证的服务目标和约束代替“更快、更稳定”；
2. 观测应能区分不同内部状态，而不只是堆积指标；
3. 每个关键状态都要对应人工或自动控制手段，诊断指标不必各自配置独立动作；
4. 重试必须有退避、随机抖动、次数上限和总时间边界；
5. 多个控制器操作同一资源时，明确优先级或分开时间尺度；
6. 依赖故障时优先隔离和降级，避免将压力转移成级联故障。

## 最小运行控制合同

自动或人工控制上线前，至少写清：

- 用户目标及不得越过的数据、安全、成本边界；
- 主观测信号的来源、负责人、正常基线和最大允许延迟；
- 每个动作的权限、最大幅度、冷却期、退出和回退条件；
- 动作后何时检查什么信号，并裁决为 `PASS`、`FAIL`、`UNKNOWN` 或 `STOPPED`；
- 进入安全状态的触发条件和当班负责人。

用户路径的实际请求结果是首要观测；服务端指标、日志和追踪用于定位原因；控制面显示的“已执行”只证明命令被接受。主信号过期、缺失或相互冲突时标记 `UNKNOWN`，暂停扩大动作。检查间隔应覆盖动作生效和指标更新的最慢时间，不得在结果可见前重复加码。

## 结果状态

| 状态 | 运行含义 | 下一动作 |
|---|---|---|
| `PASS` | 用户目标恢复，硬边界未被突破，并持续达到预设观察窗口 | 保持或逐步撤销临时控制 |
| `FAIL` | 目标未恢复，或动作造成已知负面结果 | 停止该动作，回退或换用预定动作 |
| `UNKNOWN` | 信号缺失、过期、冲突，或观察窗口未结束 | 不加码，恢复观测或人工确认 |
| `STOPPED` | 触发安全、权限、预算边界，或人工接管 | 进入安全状态，禁止自动恢复 |

默认安全状态是：停止发布和非必要自动动作，限制新流量与重试，隔离故障依赖；一致性不明时停止写入并保留数据，尽可能保留只读或状态查询能力；保存时间线和观测证据，交由明确负责人决定恢复。具体服务必须在运行手册中把这些动作映射为可执行命令，不能只写“降级”。

## 常见失败

| 现象 | 控制论诊断 |
|---|---|
| 重试风暴 | 正反馈不断放大原始故障 |
| 扩缩容反复抖动 | 阈值过近、动作过强或反馈延迟 |
| 队列持续增长 | 输入与处理能力失衡，缺少背压 |
| 有大量仪表盘但无法恢复 | 可观测但不可控制 |
| 两套自动策略相互抵消 | 多个控制器争夺同一状态 |
| 单点故障扩散 | 隔离不足，稳定余量太小 |

## 最小检查

- 用户实际感受到的目标是什么？
- 哪些内部状态能提前暴露风险？
- 每个自动动作是否有上限、冷却期和退出条件？
- 如何确认动作真正改善了用户结果？
- 单个依赖死亡时，核心能力还能保留什么？
