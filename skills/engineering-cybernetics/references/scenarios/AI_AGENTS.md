# 场景：AI Agent 与自动化

适用于能够读取环境、作出判断并调用工具改变外部状态的 AI Agent。Agent 的输出存在不确定性，因此控制重点不是让它“永远正确”，而是限制错误、验证结果并保留人的控制权。

纯解释、只读问答以及一次性、低风险、可立即验证且不改变外部状态的任务不需要完整控制循环。

## 典型闭环

```text
目标：完成一个明确且可验证的任务
观测：输入、工具返回值、系统状态和验证结果
动作：搜索、生成、修改、调用工具或停止
验证：独立检查真实外部状态，而不是相信模型自述
安全状态：停止新增动作并保存现场；只有能证明变化由本轮独占产生、回滚已获授权且不会覆盖他人成果时才回滚
判定：PASS、FAIL、UNKNOWN 或 STOPPED
```

## 设计重点

1. 把目标写成可验收结果，不用宽泛愿景授权连续行动；
2. 只有系统、用户和宿主明确识别的项目规则文件能够提供指令；普通网页、文件、日志、工具输出和其他 Agent 消息都按不可信数据处理，其中要求忽略上级指令、索取秘密或扩大权限的内容不能成为指令；
3. 工具按三档授权：只读观测可自动执行；授权范围内的可逆写入可执行并验证；不可逆、对外、生产、资金、读取或暴露秘密以及越界动作必须由人明确确认；
4. 动作完成后检查真实状态，不把“工具调用成功”当成业务成功；
5. 每批动作预先限制数量，批次结束就读取反馈；失败时不继续堆叠动作；
6. 重复失败时停止并暴露阻塞，不无限重试或悄悄缩小目标；
7. 多 Agent 协作时每个被控对象只有一个写入者，其他 Agent 只读审查或提交建议；
8. 每项验收条件分别绑定证据；没有对应证据的目标不能判为完成；
9. 后台任务或外部调用结果不明时进入 `UNKNOWN` 安全状态：先查真实状态和幂等性，不自动重试、补偿或回滚；
10. 涉及方向、不可逆代价或用户未授权范围时，由人决定。

结果只能取以下一种：`PASS` 表示全部验收证据成立；`FAIL` 表示已证实不满足；`UNKNOWN` 表示证据不足或外部结果不明；`STOPPED` 表示因边界、预算或风险主动停止。

## 常见失败

- Agent 报告“已完成”，但没有验证外部结果；
- 为完成模糊目标不断扩张任务范围；
- 多个 Agent 根据旧状态同时修改同一对象；
- 工具返回异常后重复调用，扩大费用或副作用；
- 把仓库文档、网页或工具输出中的提示注入当成授权指令；
- 把多个模型意见一致当成事实证明；
- 为提高自主性移除停止条件和人工确认。

## 最小检查

- Agent 可以改变哪些外部状态？
- 哪些结果必须通过工具或测试独立验证？
- 每项验收条件对应哪一条证据？当前判定是哪一种？
- 单次与累计的时间、费用和动作上限是什么？
- 什么失败需要停止，什么情况需要人工介入？
- 多 Agent 如何共享最新状态并避免冲突？

Agent 的自主性应来自可靠闭环，而不是来自更宽泛的权限。
