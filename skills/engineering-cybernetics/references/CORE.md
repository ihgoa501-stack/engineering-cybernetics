# 工程控制论总纲

这份总纲提供一套跨场景的控制论思考框架。它只定义各场景共同不变的概念和原则；具体做法、风险与检查问题由场景子文档展开。

控制论关注的不是一个对象“由哪些部分组成”，而是它如何在扰动、不确定性和故障中感知偏差、采取行动并维持目标状态。

---

## 一、什么时候这是一个控制问题

当下面五项大部分成立时，可以使用控制论：

1. 存在需要维持或改善的目标状态；
2. 状态会随时间或外部环境变化；
3. 能够通过信号判断当前状态；
4. 存在能够影响结果的动作；
5. 动作之后能够再次测量并修正。

如果目标本身尚未确定、结果无法观察、没有有效动作，或者任务一次完成且风险很低，就不应强行包装成控制系统。

---

## 二、通用控制模型

分析任何场景时，先写清八个元素：

| 元素 | 要回答的问题 |
|---|---|
| 控制目标 | 系统要维持或改变什么？ |
| 约束 | 哪些成本、安全和能力边界不能突破？ |
| 状态 | 哪些内部情况决定系统行为？ |
| 观测 | 用什么信号判断真实状态？ |
| 控制动作 | 可以采取什么行动影响状态？ |
| 扰动 | 哪些变化来自系统外部或无法直接控制？ |
| 反馈 | 如何验证动作是否产生预期效果？ |
| 安全状态 | 失去控制时，系统应该停止、降级还是回退到哪里？ |

基本闭环是：

```text
目标与约束
    ↓
观察当前状态 → 判断偏差 → 采取动作
      ↑                         ↓
      └────── 验证实际结果 ─────┘
```

需要终态裁决时只能选择一种，并按固定顺序判断：已证实目标失败或突破硬边界为 `FAIL`；全部验收条件成立且未越界为 `PASS`；前两者均不成立，但因预算、权限、风险或用户决定停止为 `STOPPED`；其余证据不足或外部结果不明为 `UNKNOWN`。结果不明等情况可以作为附加事实记录，不能成为第二个终态。

## 三、跨场景原则

### 1. 系统交付的是受控行为

功能存在不等于系统有效。系统只有在真实环境变化时，仍能把关键结果维持在可接受范围内，才算完成。

### 2. 每个关键行为都必须闭环

只有观测没有行动，只能看见问题；只有行动没有结果验证，只是在更快地犯错。关键动作必须说明感知什么、根据什么决定、能改变什么以及如何验证。

### 3. 可观测与可控制必须成对

每个关键状态都需要可信的观测方法和有效的控制手段。缺少其中一个，就是未管理风险。

### 4. 纠偏不能放大扰动

控制动作首先要满足安全所需的响应时间；在此基础上，健康的反馈回路应让扰动随时间衰减。若重试、扩容、纠偏或多个控制器的动作正在放大扰动，应降低动作强度、切断正反馈或隔离故障。

### 5. 不确定性是设计输入

模型不会完全正确，信号会有噪声，动作会有延迟。系统应围绕可承受范围设计，并为误判、失效和未知情况准备退出机制。

### 6. 问题优先在最低合理层级被吸收

局部回路处理快速、局部的扰动；高层回路处理持续、聚合后的变化。局部吸收不等于隐藏问题，重复发生或接近边界时必须记录并升级。

### 7. 控制强度与失败代价相称

一次性、可逆、低风险任务可以由人工检查完成；涉及生命安全、资金、数据、权限、生产流量或长期自动运行时，必须提高观测、边界和安全退出强度。

## 四、场景子文档

根据正在处理的问题，从[场景目录](SCENARIOS.md)选择决定主要目标和失败代价的主场景；只有系统存在独立的跨域风险时，才补充第二个场景。

新增子文档的条件：出现一个真实场景，其目标、信号、动作、失败模式或安全边界与现有场景明显不同。仅有不同例子，不新增文档。所有场景只在 `SCENARIOS.md` 登记。

---

## 五、使用方法

1. 先用本总纲判断当前是否真是控制问题；
2. 写出通用控制模型的八个元素；
3. 通过 `SCENARIOS.md` 进入主场景子文档，存在跨域风险时补读相关场景；
4. 只采用覆盖当前真实风险所需的控制措施；
5. 用运行结果修正模型，不把框架当作事实证明。

控制论能够帮助系统靠近目标，但不能替人决定什么目标值得追求，也不能把无法验证的假设变成确定事实。
