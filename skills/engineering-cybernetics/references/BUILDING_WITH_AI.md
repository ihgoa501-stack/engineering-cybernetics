# AI 辅助建设纪律

这份参考文档把 gstack Builder Ethos 转译为工程控制论中的建设纪律。它不替代 [ETHOS.md](../../../ETHOS.md) 和 [CORE.md](CORE.md)；当它与目标、约束、反馈或安全状态冲突时，以控制论边界为准。

## 一、它吸收什么

gstack 的核心不是“无限做大”，而是：AI 让一个人可以用流程、角色和工具获得小型软件团队级别的执行力。因此，真正稀缺的是判断、品味、范围裁决、真实验证和持续交付纪律。

本项目有意裁剪 gstack 的激进完整性口吻：只继承“当前可验证单元内不偷懒”，不继承“自动扩大范围”。是否扩大由控制目标、观测、反馈、安全状态和用户裁决决定。

本项目吸收这些部分：

1. **先找对问题**：不要把用户说出的功能直接当成目标，先确认真实用户、痛点、现有替代方案和最小可用闭环。
2. **先搜索再建设**：面对不熟悉的问题，先查已有方案、宿主能力、标准库、成熟依赖和项目内模式；搜索结果要被质疑和适配，不能直接当作答案。
3. **已选闭环内做完整**：在目标和边界已经确定后，不省略测试、错误路径、文档、评审、QA、发布验证和必要安全边界。
4. **专家流程提高观测质量**：产品、工程、设计、评审、QA、安全、发布和复盘等角色可以暴露盲点；只选择当前失败代价和验收条件需要的最小流程。
5. **真实验证收口**：代码存在、计划完成或 Agent 共识都不等于结果。需要运行、测试、截图、日志、用户行为、发布状态或其他证据。
6. **用户主权**：AI 推荐，用户决定。改变方向、扩大范围、增加成本或执行不可逆动作时，必须交给用户裁决。

## 二、它不能覆盖什么

gstack 的“完整实现”不能被解释为自动扩大范围。以下情况必须按控制论处理：

- 目标用户、问题、验收条件不清楚；
- 当前状态无法可靠观测；
- 动作后无法验证结果；
- 失败代价、权限、资金、数据或生产风险不清楚；
- 用户没有授权扩展目标、发布、付款、外部消息或不可逆操作；
- 多 Agent 只是意见一致，但没有新的实际证据。

这些情况下，正确动作是缩小范围、补观测、提出选择或进入安全状态，不是继续建设。

## 三、执行优先级

当软件、Agent、业务或自动化任务需要建设时，按下面顺序使用：

1. **控制论定边界**：目标、约束、状态、观测、动作、扰动、反馈、安全状态。
2. **用户定方向**：涉及范围、成本、风险、产品方向和不可逆动作，由用户决定。
3. **搜索定复用**：先查现有方案和项目内模式，再决定是否自建。
4. **闭环内做完整**：只在已选闭环内补齐测试、错误处理、文档、QA、评审和发布验证。
5. **真实反馈修正规则**：用运行结果和用户反馈更新场景，不为假想未来扩建规则。

## 四、和场景的关系

- 在“首个可用闭环”中，它意味着先把一个真实用户的一个流程做完整，而不是建设平台。
- 在“软件项目开发”中，它意味着计划、实现、评审、测试、发布和复盘形成连续反馈。
- 在“AI Agent 与自动化”中，它意味着多 Agent 是观测和建议通道，不是事实裁判。
- 在“业务反馈闭环”中，它意味着工程完整不能替代成交、成本、利润和用户价值证据。

一句话：gstack 负责提醒“既然决定做，就别做半成品”；工程控制论负责决定“现在到底该做什么、做到多大、如何观测、何时停止”。
