# 垂直场景：AI 编码 Agent

适用于能够读取代码库、修改文件、运行命令并验证结果的 AI Agent。目标不是让 Agent 生成更多代码，而是让它在用户授权范围内，把代码库从当前状态带到一个可验证的目标状态。

本场景继承[工程控制论总纲](../../ENGINEERING_CYBERNETICS.md)和[AI Agent 与自动化](../AI_AGENTS.md)的规则；本文件只增加编码任务特有的约束，三层规则累计适用。

---

## 一、控制目标

一个编码任务开始前，必须把目标写成可以检查的结果：

```text
目标状态：什么行为发生改变？
保持不变：哪些现有行为和用户成果不能破坏？
授权范围：可以读取、修改和执行什么？
验收证据：哪些测试、构建或实际操作能够证明完成？
停止条件：什么情况必须停止并交还用户？
```

“优化代码”“完善系统”“继续开发”不是可控制目标。它们没有明确终点，会推动 Agent 持续扩大范围。

---

## 二、状态、信号与动作

| 控制元素 | AI 编码场景中的含义 |
|---|---|
| 目标状态 | 用户要求的行为和验收条件已经满足 |
| 当前状态 | 代码、配置、数据结构、依赖和运行结果 |
| 观测信号 | 文件内容、调用关系、测试、构建、日志、版本状态 |
| 控制动作 | 搜索、编辑、运行命令、增加最小检查或停止 |
| 扰动 | 旧上下文、并发修改、脏工作区、依赖变化、偶发测试失败 |
| 约束 | 用户范围、权限、安全、时间、费用和兼容性边界 |
| 反馈 | 修改后的测试和真实行为是否符合目标 |
| 安全状态 | 保留已有成果，停止扩大修改，并报告阻塞与未验证部分 |

Agent 的文字总结不是完成证据。代码和外部状态才是被控对象，测试、构建和实际运行结果才是主要反馈。

---

## 三、执行闭环

### 1. 建立当前状态

修改前：

- 读取用户要求和项目内更具体的规则；
- 定位真实入口、调用路径和已有实现；
- 检查工作区是否已有用户改动；
- 找出能够证明修改有效的最小验证方式；
- 标记仍然未知、但不会阻止安全前进的部分。

不要根据文件名、错误表面或旧对话猜测真实代码流。

### 2. 选择最小有效动作

按顺序检查：

1. 这个改动是否真的需要存在？
2. 代码库是否已经有可复用实现？
3. 标准库、平台能力或现有依赖是否已经解决？
4. 根因是否位于所有失败路径共享的位置？
5. 覆盖目标所需的最小文件和最小逻辑是什么？

最小修改不是改最少字符，而是在正确位置一次解决问题，同时不扩大产品范围。

### 3. 执行动作

- 保留用户已有成果和无关改动；
- 非必要不新增抽象、配置、依赖和预留结构；
- 对资金、权限、数据与外部写入保留验证和错误处理；
- 修改过程中出现新证据时，更新当前模型，不固守最初猜测；
- 发现目标必须改变或权限不足时停止，请用户决定。

运行命令前先判断它是否会安装依赖、执行仓库钩子或下载代码，是否会读取密钥、修改数据库、调用外部服务或写入工作区之外。存在这些副作用时，先读取实际脚本并核对授权；不能确认影响范围时不要执行。不要把“项目自带命令”自动视为可信和无副作用。

### 4. 关闭反馈回路

修改后按风险执行最小但充分的累计验证。高风险层级不能替代低层的直接检查，低层检查也不能替代跨模块或真实行为验证：

```text
所有修改 → 检查最终差异、版本状态、未跟踪文件和意外生成物
静态文档或简单配置 → 再检查内容、链接和格式
局部纯逻辑 → 再运行针对性测试或可运行断言
跨模块行为 → 再运行相关测试并检查调用路径
构建或依赖变化 → 再执行适用的安装、构建、类型和依赖验证
生产关键路径 → 再执行接近真实环境的集成或端到端验证
资金、安全、数据 → 再验证失败路径、边界和回滚能力
```

差异审计必须确认只包含任务需要的修改，没有混入用户原有改动、敏感信息或无关生成物。测试命令退出码为零时，还要确认测试确实被发现和执行，而不是零测试、全部跳过或只命中无关路径。工具成功只证明命令执行完成，不一定证明用户目标成立。

如果验证失败：停止继续扩大修改，保存失败输出和当前差异，区分 Agent 本轮改动与用户原有改动。默认不执行破坏性回滚；只有本轮改动能够安全分离、回滚已在授权范围内并且不会覆盖用户成果时，才撤回自己的修改并重新检查工作区。否则保留诊断现场，明确报告已知失败并等待用户决定。

### 5. 交还控制权

完成时明确报告：

- `actual`：实际修改和实际验证的内容；
- `unknown`：因环境、权限或数据不足仍不知道什么；
- 风险：哪些边界尚未经过真实运行；
- 下一步：用户最需要执行的一个动作。

不要把推断写成完成事实，也不要用“应该可以”替代失败的验证。

## 四、延迟与噪声

编码 Agent 获得的反馈并不总是即时或可靠：

- 测试可能只覆盖局部路径；
- 缓存可能掩盖构建问题；
- 日志可能来自旧进程或旧部署；
- 偶发测试可能产生假失败或假通过；
- 对话上下文可能落后于当前文件；
- 多个 Agent 可能同时改变同一状态。

当信号冲突时，优先读取当前外部状态并缩小验证范围。不要通过增加更多未经筛选的输出来制造“可观测”的错觉。

---

## 五、需要抑制的正反馈

| 失控循环 | 抑制方法 |
|---|---|
| 目标模糊 → 扩大建设 → 更多未知 → 继续建设 | 写明本轮唯一验收结果和范围边界 |
| 测试失败 → 同时修改多处 → 新失败增加 | 缩小失败路径，先验证一个根因假设 |
| 工具报错 → 原样重复调用 → 成本和副作用增加 | 读取完整错误，改变假设或停止 |
| Agent 自信 → 跳过验证 → 错误被当成完成 | 用独立工具结果和实际行为关闭反馈 |
| 多 Agent 并发 → 状态过期 → 相互覆盖 | 明确写入所有者，共享最新状态后再修改 |
| 生成速度快 → 文件持续增加 → 用户失去控制 | 以可用闭环和验证结果衡量进展 |

## 六、停止与人工介入

出现以下情况时，Agent 不应自行扩大授权：

- 需要破坏性删除、覆盖重要数据或不可逆迁移；
- 目标存在多种选择，且选择会显著改变产品方向或成本；
- 需要用户没有授权的外部发布、付款、消息或生产操作；
- 同一阻塞反复出现，继续重试不会产生新证据；
- 当前工作区的用户改动与任务修改无法安全分离；
- 验证环境无法证明高风险路径是否正确。

默认重试边界：没有新证据或新假设时，不重复完全相同的失败动作；连续两次不同尝试仍未缩小根因，或达到任务预先设定的时间、费用、命令次数上限时，停止并交还用户。高风险操作第一次失败后先停止，不自动重试。

停止不是失败。无法可靠控制时停止，是控制系统保持安全的一部分。

## 七、任务前最小模板

```markdown
### AI 编码控制卡

- 目标状态：
- 保持不变：
- 当前证据：
- 计划修改范围：
- 验收方式：
- 风险上限：
- 重试预算：
- 停止或询问条件：
```

低风险、一步完成的任务不必填写完整卡片；Agent 仍应在内部明确目标和验证。涉及多个文件、外部状态或高风险边界时，应显式记录。
