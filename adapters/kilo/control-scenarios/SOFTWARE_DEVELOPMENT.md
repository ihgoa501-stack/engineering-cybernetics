# 场景：软件项目开发

适用于把一个真实问题转化为可运行、可验证并能交给用户使用的软件项目。它控制的是项目从问题到结果的推进过程，不是一次编码任务，也不是软件上线后的运行稳定性。

| 相邻场景 | 控制对象 |
|---|---|
| 软件项目开发 | 项目是否走到真实可用结果 |
| [AI 编码 Agent](ai-agents/CODING_AGENT.md) | 一次编码任务是否正确完成 |
| [软件系统](SOFTWARE_SYSTEMS.md) | 上线后的系统是否稳定运行 |

## 典型闭环

```text
确认真实问题
    ↓
限定本轮唯一闭环与资源上限
    ↓
完成可运行版本
    ↓
验证并交给真实用户
    ↓
观察使用结果
    ↓
停止、修正、重复验证或小规模扩大
```

## 设计重点

1. 用用户行为和实际结果定义目标，不用功能数量或代码量定义进展；
2. 每轮只选择一个可以从起点走到结果的端到端流程；
3. 调查、开发、验证和真实使用设置明确阶段门，不以文档或模型信心代替证据；
4. 新想法默认进入后续清单，只有完成当前闭环必需的内容才能进入本轮；
5. 时间、资金和复杂度达到上限时，缩小范围或停止，不用继续建设掩盖未知；
6. 第一次结果只判断本轮假设，重复验证后才讨论可重复性；
7. 用户反馈决定下一步，不在没有使用证据时自动扩建相邻功能。

## 可观测状态

- 用户和问题是否具体；
- 用户当前怎样解决问题；
- 本轮端到端流程是否明确；
- 核心路径当前在哪一步中断；
- 验收证据是否真实运行；
- 是否已有目标用户实际使用；
- 时间、费用和复杂度是否接近上限；
- 当前最关键的 `unknown` 是否正在被消除。

## 常见失败

| 现象 | 控制论诊断 |
|---|---|
| 持续增加功能但核心流程不可用 | 控制目标被功能数量替代 |
| 研究越来越深却没有用户结果 | 开环探索，没有交付反馈 |
| AI 快速生成大量代码 | 动作增益过高，人的可观测与控制能力下降 |
| 同时解决多个用户和流程 | 多个控制目标相互争夺资源 |
| 测试通过但无人真实使用 | 技术反馈被误当成用户反馈 |
| 一次成功后立即扩大范围 | 根据噪声过度纠偏，缺少重复验证 |

## 最小检查

- 具体是谁在经历什么问题？
- 他现在怎样解决？
- 本轮唯一端到端流程是什么？
- 什么结果证明它可以真实使用？
- 时间、资金和范围上限是什么？
- 当前最关键的未知是什么？
- 什么证据允许继续、修改、停止或扩大？
